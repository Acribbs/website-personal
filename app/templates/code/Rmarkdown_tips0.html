Rmarkdown Tips

  • Adam Cribbs
  • February 19, 2019

I use Rmarkdown almost every day and have been an intense user for the past 2 years.

I primarily use Rmarkdown to build and share next-generation sequencing reports for collaborators or analysis code to my students or postdocs in the group. Having made many errors and written lots of inefficient R code I would like to share some of my Rmarkdown tips that have helped me in the past

Tip1: Use R projects

If you use Rstudio to write your Rmarkdown code then setting projects is super useful way organising your code. If you work on many different projects then it can be a pain to set working directories and organise your data in a way that makes it easy to change between projects easily.

To set up an R project its as easy as selecting File , New Project... then to switch between projects you can use the top right projects tab and switch easily

Tip2: Set global options for all code chunks

Copy and pasting options for every chunk can be incredibly annoying and time wasting. Its better to set the code options in the first chunk. This can be useful when writing sequencing analysis reports for both non-computationally minded people, where code is distracting, and computational people, where code is useful. For example, the echo=TRUE/FALSE can easily modify whether code will be displayed or not when knitting. Adding this option for all of your code chunks can be time consuming. Thankfully, there is a better way - you can specify knitr::set_opts(echo=TRUE/FALSE) and it will apply that option to all chunks. For example: 

	
		```{r setup, include=FALSE}
		knitr::opts_chunk$set(echo=TRUE)
		```

Tip3: Name all of your code chunks

You can name all code chunks, which can be quite time consuming when you have a huge Rmarkdown document, particularly since each code chunk needs to have a unique name.

	
		```{r name_of_code_chunk}
		some code here
		```

However, there are some quite clear advantages to naming. For example, 1. It really helps with debugging, as the code chunk can be easily identified by the name. 2. If you have cacheing activated then there isn't a slowdown. 3. It makes your code documentation easier to understand as you can give informative titles that can be used to remind you of what you did.

Tip4: Reduce the knit time

If you set cache=TRUE in your chunk options or in the knitr::opts_chunk$set(), R will cache your output (save the data) so they wont be run every time you knit your Rmarkdown. This is very handy if you have a piece of code that you dont want to run every time, particularly if it takes a long time to run. If you do change something in your code then R will realise and then re-run the code chunk. However, this tip can also be confusing and annoying at times because it can sometimes lead to unexpected behaviour and unusual errors that can be difficult to trace. As a work around, I would only set cache=TRUE when you have finished writing your code. Always check the output following the final knitting and make sure that the output is what you expect.
